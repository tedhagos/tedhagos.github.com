Abstraction and Complexity

Complexity can be managed by a system of hierarchic abstractions. One layer on top of the other. Each layer hides some degree of implementation from the layer on top. One layer of abstraction building on top of other abstractions. Each understandable by itself.

There are factors that drive the complexity of system

Complexity of the problem domain
Difficulty in managing the software development process
Flexibility is possible through software
Software is a discrete system

Abstraction defines the essential characteristics of an object. It makes the object different from any other object.

The elevator is different from an escalator—how? go figure. The essence of an object makes it distinguishable from other objects in the eye of the obsever (you).

It allows an observer to appreciate object without knowing how it works internally. Like the light switch, you really don’t know how electricity works but you know you can get the light if you flip the switch. If you flip it in the opposite direction, light is off.

Abstraction is one of the major focus of OO analysis and design. It allows you to ignore the details of an object. It simply presents the protocols (big word) of interaction between objects — in Java, the unit of abstraction is the class/object.

Abstraction helps us grapple with large complex systems by chunking. The span of absolute judgement and span of immediate memory impose limitations on our capability to understand things, concepts and procedures. To break the information bottleneck, we chunk into several dimensions.

We focus on one chunk and ignore the inessential details of others. We choose to deal with the generalized and idealized model of the object in front of us. This way, we can increase its semantic content
